# Feature Implementation: Employee Creation

## Status

Completed

## Module

Users

## Goal

Allow an authorized administrator to create an employee record with a unique employee ID and valid employment information.

## User Story

As an administrator, I want to create an employee record so that the employee can be managed within the system.

## Preconditions

* The administrator must be authenticated.
* The administrator must have permission to create employees.
* The related user account must exist where required.
* The employee must not already exist.

## Inputs

* User ID
* First name
* Last name
* Email address
* Telephone number
* Position
* Employment start date
* Employment type
* Assigned client
* Initial employment status

## Business Rules

* Employee ID must be unique.
* A user cannot have more than one active employee record.
* Employment start date cannot be invalid.
* Required fields must be present.
* Initial status must be one of the supported employee statuses.
* Employee creation must be atomic.

## Request Flow

```text
POST /api/employees/
        ↓
EmployeeCreateView
        ↓
EmployeeCreateSerializer
        ↓
EmployeeService.create()
        ↓
EmployeeValidator.validate_create()
        ↓
EmployeeRepository.create()
        ↓
Database
```

## API Contract

### Endpoint

```http
POST /api/employees/
```

### Request

```json
{
  "user_id": 15,
  "position": "Software Developer",
  "employment_start_date": "2026-07-12",
  "employment_type": "FULL_TIME",
  "client_id": 4
}
```

### Success Response

```json
{
  "id": 21,
  "employee_id": "EMP-000021",
  "user_id": 15,
  "position": "Software Developer",
  "employment_start_date": "2026-07-12",
  "employment_type": "FULL_TIME",
  "status": "ACTIVE"
}
```

### Error Responses

```text
400 — Invalid input
403 — Insufficient permission
404 — Related user or client not found
409 — Employee already exists
```

## Decisions

### Employee ID Generation

Employee IDs will be generated by a dedicated ID-generation service rather than directly inside the model.

Reason:

* supports multiple ID formats;
* keeps ID-generation logic reusable;
* avoids coupling the employee model to one generation strategy.
